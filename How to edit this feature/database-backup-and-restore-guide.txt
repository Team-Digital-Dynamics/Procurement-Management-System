Title:
Add database backup and restore support

Description:
Implemented database integrity features for manual backup, scheduled backup, and backup restore.

Changes included:
- Added admin-only backup endpoints under /api/backups.
- Added manual database backup creation with SQL export and SHA-256 checksum file.
- Added scheduled daily backup job, configurable through application.yml or environment variables.
- Added restore support that verifies checksum before restoring data.
- Added audit logging for manual backup, scheduled backup, and restore actions.
- Added backup configuration properties for directory, cron schedule, and scheduler enablement.
- Added integration test coverage for manual backup, scheduled backup, and restore integrity.

Acceptance results:
- Manual backup: Successful
- Scheduled backup: Successful
- Restore backup: Successful

How it works:

Manual backup:
1. An admin sends a POST request to /api/backups/manual.
2. BackupController calls DatabaseBackupService.createBackup("MANUAL").
3. DatabaseBackupService reads application database tables.
4. It writes DELETE and INSERT SQL statements into a timestamped .sql file.
5. It creates a SHA-256 checksum file next to the backup.
6. The controller records an audit log entry.
7. The API returns the backup file name, checksum, table count, row count, and timestamp.

Scheduled backup:
1. Spring scheduling runs DatabaseBackupJob using the configured cron expression.
2. The default schedule is 02:00 every day.
3. The job checks pms.backup.scheduled-enabled.
4. If enabled, it calls DatabaseBackupService.createBackup("SCHEDULED").
5. A .sql backup file and .sha256 checksum file are created.
6. The system records an audit log entry with actor SYSTEM.

Restore backup:
1. An admin sends a POST request to /api/backups/restore.
2. The request body must include the backup file name.
3. Example:
   {
     "fileName": "pms-backup-20260708-110342-123.sql"
   }
4. DatabaseBackupService verifies that the file is inside the configured backup directory.
5. It reads the matching .sha256 checksum file.
6. It recalculates the backup file checksum.
7. If the checksum does not match, restore is rejected.
8. If the checksum matches, restore starts.
9. Referential integrity is temporarily disabled for the restore.
10. The backup SQL statements are executed.
11. Referential integrity is enabled again.
12. The controller records an audit log entry.
13. The API returns the restored file name, checksum, timestamp, and executed statement count.


API summary:

List backups:
GET /api/backups

Create manual backup:
POST /api/backups/manual

Restore backup:
POST /api/backups/restore

Restore request body:
{
  "fileName": "pms-backup-20260708-110342-123.sql"
}


Configuration summary:

Default configuration in application.yml:
pms:
  backup:
    directory: ${PMS_BACKUP_DIRECTORY:backups}
    cron: ${PMS_BACKUP_CRON:0 0 2 * * *}
    scheduled-enabled: ${PMS_BACKUP_SCHEDULED_ENABLED:true}

To change the backup folder:
Set PMS_BACKUP_DIRECTORY to the folder path where backup files should be stored.

To change the backup schedule:
Set PMS_BACKUP_CRON to a valid Spring cron expression.

To disable scheduled backups:
Set PMS_BACKUP_SCHEDULED_ENABLED=false.
